# Расчёт остатков товаров (Kotlin)

Консольная программа: читает CSV со списком операций (поступления и продажи), последовательно
пересчитывает остатки и формирует итоговый CSV с остатком после каждого движения товара.

## Запуск

Нужен JDK 17+ (проверено на 17 и 25), Maven не обязателен — в репозитории есть wrapper.

```bash
# сборка + тесты, на выходе исполняемый jar
./mvnw clean package

# запуск: два аргумента — входной и итоговый CSV
java -jar target/stock-calculator.jar samples/operations.csv result.csv
```

Обязательных переменных окружения нет. Уровень логирования (SLF4J + Logback, по умолчанию `INFO`)
задаётся системным свойством:

```bash
java -Dlog.level=DEBUG -jar target/stock-calculator.jar samples/operations.csv result.csv
```

Только тесты — `./mvnw test`: 18 тестов покрывают логику продаж по рангам, разбор CSV
и воспроизводимость примеров из `samples/`.

## Формат входного файла

Разделитель — `;`, тип операции определяется числом колонок:

| Операция    | Формат                                 | Пример            |
|-------------|----------------------------------------|-------------------|
| Поступление | `<id группы>;<id товара>;<количество>` | `G1;IPHONE-15;10` |
| Продажа     | `<id группы>;<количество>`             | `G1;12`           |

## Формат выходного файла

Одна строка на каждое движение товара. Продажа из группы может затронуть несколько товаров,
поэтому одна операция может дать несколько строк — их объединяет общий `operation_number`.

| Колонка            | Значение                                                           |
|--------------------|--------------------------------------------------------------------|
| `operation_number` | порядковый номер операции (с 1)                                    |
| `type`       | `RECEIPT` — поступление, `SALE` — продажа                          |
| `group_id`   | id группы                                                          |
| `product_id` | id товара (пустой — недостача по группе, см. допущение 4)          |
| `quantity`   | количество со знаком: `+` поступление, `−` продажа                 |
| `balance`    | остаток **этого товара** после движения (может быть отрицательным) |

Готовый пример CSV-файла, который формируется на выходе, лежит в репозитории:
[`samples/result.csv`](samples/result.csv) — он получен программой из входного
[`samples/operations.csv`](samples/operations.csv):

```csv
operation_number;type;group_id;product_id;quantity;balance
1;RECEIPT;G1;IPHONE-15;10;10
2;RECEIPT;G1;IPHONE-14;5;5
3;SALE;G1;IPHONE-14;-3;2
4;SALE;G1;IPHONE-14;-2;0
4;SALE;G1;IPHONE-15;-10;0
5;RECEIPT;G2;TV-SONY;7;7
6;SALE;G2;TV-SONY;-10;-3
7;RECEIPT;G1;IPHONE-13;20;20
8;SALE;G1;IPHONE-13;-6;14
9;SALE;G3;;-4;-4
```

Второй пример с краевыми случаями (заголовок, комментарии, битые строки, ранг `A10 < A9`) —
[`samples/operations-edge-cases.csv`](samples/operations-edge-cases.csv) и его результат
[`samples/result-edge-cases.csv`](samples/result-edge-cases.csv).

## Принятые допущения

Неоднозначности условия решены так:

1. Наивысший ранг — лексикографически **первый** `id товара`; сравнение строковое: `A10` выше `A9`.
2. Итоговый CSV — строка на каждое движение товара: продажа из группы может списать несколько
   товаров, такие строки делят общий `operation_number`.
3. Продажа сверх остатка: продаётся всё доступное, недостача уходит в минус по низшему
   по рангу товару группы.
4. Продажа из группы без поступлений: минус пишется на псевдо-товар с пустым `product_id`;
   поступления реальных товаров его не гасят.
5. Товар с остатком ≤ 0 при продаже пропускается; минус гасится только поступлением того же товара.
6. Некорректные строки логируются с `WARN` и пропускаются.
7. Заголовок необязателен, пустые строки и `#`-комментарии игнорируются, кодировка UTF-8
   (BOM допускается), количества — целые.
